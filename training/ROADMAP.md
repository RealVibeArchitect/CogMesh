# 멀티모달 확장 로드맵 — 텍스트 → 이미지 → 동영상

태양님 전략: "작은 텍스트 모델부터 GPU로 학습시키고 확장한다."
모든 단계가 **같은 PAD 출력 공간 (P, A, D)** 을 공유하므로, 인코더만 갈아끼우면 된다.

---

## ✅ 1단계 — 텍스트 → PAD  (지금, 로컬 GPU에서 진짜 학습 O)

- 이 저장소가 구현한 것.
- 다국어 MiniLM + LoRA + 회귀 헤드.
- `python scripts/train.py` 로 VRAM 자동 감지 후 실제 학습 (6GB부터 가능).

**확장 준비**: 데이터를 늘릴수록 좋아진다. `data/seed_emotions.jsonl`에 줄 추가.

---

## 🟡 2단계 — 이미지 → PAD  (저VRAM도 가능, 단 사전학습 재사용)

- **핵심**: 이미지 인코더를 처음부터 학습하지 않는다 (저VRAM으로는 무리).
  사전학습된 CLIP 비전 인코더를 **얼리고(freeze)**, 그 위에 PAD 회귀 헤드만 학습.
- 구조:
  ```
  이미지 → [CLIP ViT (frozen)] → 임베딩 → [PAD 헤드 (학습)] → (p,a,d)
  ```
- 헤드만 학습하므로 메모리 매우 적게 든다. 6GB급에서도 충분.
- 데이터: (이미지, PAD 라벨) 쌍 필요. 표정 데이터셋 + 감정 라벨 활용 가능.

**정직한 주의**: CLIP 임베딩이 "감정"을 얼마나 담는지가 성능을 좌우한다.
표정/분위기는 어느 정도 되지만, 완벽하지 않다. 실험이 필요하다.

---

## 🔴 3단계 — 동영상 → PAD  (저VRAM으로는 벅참, 정직하게)

- 동영상 = 프레임 여러 장 → 메모리가 프레임 수에 비례해 폭발.
- **저VRAM(예: 6GB)으로 동영상 인코더 학습은 현실적으로 어렵다.** 두 가지 우회:
  1. **프레임 샘플링 + 2단계 재사용**: 동영상에서 몇 프레임만 뽑아 각각 이미지→PAD로
     처리하고 시계열로 평균/집계. (학습 없이 2단계 모델 재사용 → 저VRAM도 가능!)
  2. **클라우드 GPU** (A100 등)를 빌려 진짜 비디오 인코더 학습. (비용 발생)
- 추천: 먼저 **우회 1**로 "동영상도 되긴 한다"를 확인하고, 필요하면 클라우드로.

---

## 🧩 통합 지점

세 인코더 모두 `(p, a, d)`를 출력하므로, 웹앱은 어느 모달리티든 동일하게 받는다:

```
텍스트 ─┐
이미지 ─┼─→ (p, a, d) ─→ PADState.update() ─→ 메타인지 ─→ 입력 변환 (Sprint 20)
동영상 ─┘
```

즉 Sprint 11~20에서 만든 웹앱 인지 파이프라인은 **하나도 안 바꿔도** 된다.
입력 소스만 늘어날 뿐. 이게 "같은 출력 공간"을 고집한 이유다.
