# PAD Encoder — 텍스트를 인지(PAD 좌표)로 변환하는 학습 모델

CogMesh 프로젝트의 **진짜 학습(GPU) 경로**. 웹앱(causal_chat_v4)에서 못 하는
신경망 파라미터 학습을 로컬 GPU에서 수행한다. 6GB 노트북 GPU부터 24GB 데스크톱
카드, 80GB A100까지 — VRAM을 자동 감지해 설정을 맞춘다.

## 🎯 목표

입력 텍스트를 **PAD 인지 좌표 (P, A, D) ∈ [-1, +1]³** 로 변환하는 인코더를 학습한다.

```
"드디어 목표를 달성했어!"  →  P:+0.9, A:+0.7, D:+0.8  (환희 근처)
"이거 실패하면 어쩌지..."   →  P:-0.3, A:+0.7, D:-0.4  (긴장 근처)
```

이 출력은 웹앱 `src/core/pad/`의 좌표계와 **정확히 동일**하다.
따라서 학습된 인코더는 웹앱의 규칙 기반 PAD를 대체/보강할 수 있다.

## 🖥️ GPU 자동 적응 (VRAM 자동 감지)

**아무 CUDA GPU에서나 돌아가게** 설계했다. 시작 시 `train.py`가 VRAM을 감지해
batch size, max_length, gradient accumulation, gradient checkpointing을 자동으로
맞춘다 (`utils.auto_tune`). config에 값을 직접 적으면 그 값이 우선한다.

| VRAM | 자동 batch | max_length | 비고 |
|------|-----------|-----------|------|
| ~6GB (노트북 RTX) | 8 | 64 | 기존 안전 프로파일 |
| ~10–12GB | 16 | 96 | 중급 |
| ~16–24GB | 32 | 128 | 대형 배치 |
| 32GB+ (A100/H100) | 64 | 192 | 초대형 배치 |

공통 설계:
- 작은 베이스 모델 (기본: 다국어 MiniLM ~120M) — VRAM 여유가 크면 더 큰 모델로 교체 가능
- **LoRA** (전체 파인튜닝 X, 어댑터만 학습 → 메모리 절감). VRAM 넉넉하면 `use_lora: false`로 풀 파인튜닝
- **4-bit 양자화** (bitsandbytes) — *향후 계획* (현재 코드 미구현; MiniLM 크기에선 LoRA만으로 충분)
- 회귀 헤드 (3개 실수 출력: P, A, D)
- mixed precision(AMP)으로 VRAM 절반

## 🧩 왜 텍스트부터인가 (태양님 전략)

큰 걸 한 번에 하지 않는다. 텍스트 인코더를 단단히 만들고 확장한다:

```
1단계 (지금): 텍스트 → PAD          ✅ 로컬 GPU에서 진짜 학습
2단계:        이미지 → PAD          (사전학습 CLIP 재사용, 가볍게)
3단계:        동영상 → PAD          (프레임 샘플링 + 클라우드 GPU 고려)
```

모든 단계가 **같은 PAD 출력 공간**을 공유하므로, 인코더만 바꿔 끼우면 된다.

## 📁 구조

```
pad_encoder/
├── configs/          학습 설정 (YAML)
├── data/             학습 데이터 (텍스트 + PAD 라벨)
├── src/              모델 / 데이터셋 / 학습 코드
├── scripts/          실행 스크립트 (train, infer, export)
└── checkpoints/      학습된 가중치 저장
```

## 🚀 빠른 시작 (로컬 GPU)

```bash
# 1. 환경 설치
pip install -r requirements.txt

# 2. 샘플 데이터로 학습 (진짜 학습 전 파이프라인 점검)
python scripts/train.py --config configs/auto.yaml

# 3. 추론 테스트
python scripts/infer.py --text "드디어 해냈어!"

# 4. 웹앱 연동용으로 내보내기 (선택)
python scripts/export_onnx.py
```

## ⚠️ 정직한 범위

- 이 저장소는 **텍스트→PAD 학습**을 완전히 실제로 수행한다 (아무 CUDA GPU나 가능).
- 이미지/동영상은 **구조적 자리(인터페이스)만** 마련하고, 실제 학습은 다음 단계.
- 학습 품질은 **데이터 품질에 달려 있다** — 좋은 (텍스트, PAD) 쌍이 많을수록 좋다.


## 🆕 이번 파이프라인 개선 (검증 완료)

실제로 굴러가는 GPU 학습 파이프라인으로 다듬었다:
- **최신 `torch.amp` API** 로 교체 (deprecated 경고 제거)
- **코사인 LR 스케줄러 + 웜업**, **gradient clipping**, **early stopping**
- **grad_accum 잔여 배치 flush** 버그 수정 (마지막 배치 손실 방지)
- **감정 층화(stratified) 분할** — val이 20개 감정을 모두 포함
- **축별(P/A/D) MAE + within-0.2 정확도** 지표로 어느 축이 약한지 진단
- **`pad_encoder_best.pt` / `pad_encoder_last.pt`** + **`history.json`** 저장, `--resume` 지원
- **`export_onnx.py` 신규** — LoRA 병합 후 ONNX 내보내기 + onnxruntime 검증
- **씨앗 데이터 40 → 200개** (감정당 10문장, 다양한 한국어 표현)
- `src/utils.py` 로 시드/지표/체크포인트 로더 공용화 (infer·serve·export 공유)

> 파이프라인 로직(학습 루프·스케줄러·층화분할·지표·체크포인트 왕복)은
> 스텁 인코더로 **실제 실행 검증**을 마쳤다. 남은 건 로컬 GPU에서
> `pip install -r requirements.txt` 후 `python scripts/train.py` 뿐이다.

## 🆕 데이터 보강 v2 (200 → 254개)

첫 학습(초기 개발 GPU였던 RTX 4050에서 실제 완주, val_mse=0.1614) 후 관찰된 두 가지 문제를 데이터로 고쳤다:

- **"흥분(excited)"과 "낙관(optimistic)"이 A(각성도)축에서 헷갈림**
  → 두 감정을 명확히 대비시키는 문장 각 5개씩 추가 (excited=즉각적 두근거림 A 높음,
  optimistic=차분한 미래 기대 A 낮음)
- **평범한 문장("안녕", "밥 먹었어?")도 감정으로 과잉 해석**
  → `neutral` 라벨(P=A=D=0) 20개 추가, 모델이 "짧으면 다 긍정"으로 뭉개지 않게
- **감정마다 앵커 좌표 하나만 배우는 문제** (문장 10개가 전부 같은 PAD값)
  → 대표 감정 8개에 강/중/약 3단계 강도 문장 추가 → 좌표가 0으로 수렴하는 연속값 학습

`stratified_split`으로 새로 생긴 21개 클래스(neutral 포함)도 전부 val에 포함되는 것 확인함.

## 🆕 데이터 보강 v3 (254 → 332개) — neutral 부작용 해소

v2 학습 후 **"나 행복해"가 지루함(bored)으로 예측되는** 참사 발견. 원인 진단:
- 10자 이하 짧은 문장을 **neutral이 100% 독점** → 모델이 "짧다 = 무감정"으로 오학습
- 학습 데이터에 **"행복" 단어가 사실상 없음** (전부 "기쁘다/최고다"로만 표현)

수정:
- **짧고 강한 감정 문장 대량 추가** ("나 행복해", "너무 슬퍼", "긴장돼" 등) → 짧은 문장의
  neutral 비중 100% → 20%로 감소, 14개 감정이 짧은 문장에 고루 분포
- **일상 감정 단어 채우기** — 행복/사랑/좋아/싫어 계열 (elated·disgust 등에 배치)
- **neutral 재정비** — 감정 단어 배제하고 '사실 확인 질문'("이 파일 어디 저장했어?")에 집중

결과: 총 332개, "행복" 포함 문장 2 → 7개, 짧은 문장 감정 다양성 확보.
